# Session: Node 1 & 5 not recording — cf_clearance staleness

## Problem
Nodes 1 and 5 (NL-proxied nodes) were not recording channels despite the cookie refresher successfully obtaining fresh cf_clearances through the socks-to-http-proxy.

## Root Cause
Startup order bug in `main.go`:

1. `loadDotEnv(".env")` (line 88) loads stale cf_clearance from MINI_ENV secret
2. `config.New(c)` (line 343) reads env vars → `Config.CfClearance` is set to OLD value
3. `LoadSettings()` from Supabase (line 352) checks guard `Config.CfClearance == ""` → **skips overwrite** → DVR uses stale clearance

The cookie refresher saved a fresh cf_clearance to Supabase, but the guard in `LoadSettings()` (server/config.go:76,85) prevented it from being loaded because the .env already populated the field.

## Fix
Zero out cookie-related Config fields before the initial `LoadSettings()` call in `main.go:351-357`:
- `Config.Cookies = ""`
- `Config.CfClearance = ""`
- `Config.SessionID = ""`
- `Config.Csrftoken = ""`

The guard is still active for the 5-minute periodic reload (line 404), which is correct — it prevents a fresh in-memory clearance from being clobbered by stale Supabase data during a race with extractCfClearance.

## Other Findings
- Both NL proxies (51.159.226.80:1080 for node-1, 144.124.232.204:1080 for node-5) work — cookie refreshers succeeded through them
- Node-1: Cloudflare found Turnstile → solved it (~30s)
- Node-5: "ERROR: No Cloudflare challenge found" → no challenge needed, clearance still valid
- The RDP browser shows "Arizona" because it connects DIRECTLY (not through proxy) — the DVR routes through the NL proxy
- DVR started with `Chrome/146` user-agent (hardcoded fallback in .env), not `Chrome/148` from the browser
- Node-5 cookie refresher saved to `dvr_settings_4` — NODE_ID may be intentionally set to "4" on node-5 repository
- Runs were ~20-40 min in before cancellation; DVR logs (dvr-output.log) unreachable after VM termination
